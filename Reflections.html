Using History for Efficiency

How could having easy access to the entire history of a file make you a more efficient programmer in the long term?

Changes that introduced errors in later changes can be identified.
Changes can suggest coding patterns that should be avoided or should continue to be implemented.

Manual Commits
What do you think are the pros and cons of manually choosing when to create a commit, like you do in Git, vs having versions automatically saved, like Google docs does?
Pros - Better able to identify logical commit points, Easier to identify from shorter list of changes with descriptions.
Cons - Risk of missing a logical commit event before an error is introduced, Risk of missing a commit event due to system or network failure.


Multi-File Commits
Why do you think some version control systems, like Git, allow saving multiple files in one commit, while others, like Google Docs, treat each file separately?
Git is project oriented, Google Docs is single file oriented

Using Git to View History
How can you use the commands git log and git diff to view the history of files?
git log - Shows history of changes with commit ids, optionally with statistics
git diff - Shows the differences between the commits, given by two commit id parameters

Confidence and Version Control
How might using version control make you more confident to make changes that could break something?
I can always revert to a place before the error.

How Do You Want to Use Git
Now that you have your workspace set up, what do you want to try using Git for?
Custom Wordpress plugin and theme development


Staging Area
How is the staging area different from the working directory and the repository? What value do you think it offers?
The working directory is an actual folder and the staging area is a virtual container.  The staging area provides a place to collect logically common files.

Visualizing with Diagrams
How do the diagrams help you visualize the branch structure?
They identify the current commit and which commits are and are not attached to it.

What is the result of merging two branches together? Why do we represent it in the diagram the way we do?
It combines the changes in the different branches into a single branch.
It indicates the inheritance of the changes.

Why might you want to always pull changes manually rather than having Git automatically stay up-to-date with your remote repository?
So you don't have unexpected changes showing up.

Describe the differences between forks, clones, and branches. When would you use one instead of another?
Forks - Duplicates repository on GitHub, use when you do not want to share updates with originator.
Clones - Duplicates repo on either GitHub or local systems
Branches - Names and catalogs a set of changes

What is the benefit of having a copy of the last known state of the remote stored locally?
Offline access of most current commits.

How would you collaborate without using Git or GitHub? What would be easier, and what would be harder?
From experience, email and shared documents are much harder, but organization policies can get in the way of easy.

When would you want to make changes in a separate branch rather than directly in master? What benefits does each approach have?
When you are experimenting with a new idea.  Separate branches allow you to operate on multiple ideas, individually, or operate on isolated fixed, when working with a team.
